# Feature Specification: Arbiter Workbench UI

**Feature Branch**: `003-arbiter-workbench-ui`  
**Created**: 2026-05-13  
**Status**: Draft  
**Input**: User goal: "User starts from Markdown in the UI, clicks LLM-assisted parsing to generate a draft, experts revise the draft into a reviewed version, and export an integrated package."

Arbiter Workbench UI is an Admin-side expert review workbench for regulation
and internal-policy structuring. The revised MVP starts from Markdown
regulation text inside the UI, invokes the 001 Regulation Structuring Pipeline
through an Admin-only structuring adapter, displays the LLM-assisted
`StructuringPipelineOutput` draft, lets experts revise and approve draft fields,
and exports an integrated reviewed-for-structuring package.

The workbench MUST NOT call model providers directly. LLM-assisted parsing is
performed only by the 001 pipeline or a thin Admin adapter that delegates to
001 and routes model calls through `LLMClient / ModelProvider`. All generated
assets remain draft or reviewed-for-structuring Admin artifacts; they are not
active `RulePack`, formal `RuleItem`, runtime-safe reviewed assets, final
`JudgmentResult`, or compliance conclusions.

Runtime-facing 002 scenario and trace screens remain future/mock-only and are
outside the revised MVP acceptance path.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - LLM-Assisted Markdown Structuring Review (Priority: P1)

An expert pastes or uploads Markdown regulation text, chooses whether to use
LLM-assisted parsing, clicks the parse action, receives a 001
`StructuringPipelineOutput` draft, reviews
document metadata, unit tree, semantic draft fields, reference candidates,
dependency edges, validation findings, and extraction provenance, edits draft
fields, records review decisions, and exports an integrated package marked
`reviewed_for_structuring` only when required expert review gates are satisfied.

**Why this priority**: This is the core workbench value path. The user should
not need to manually load generated JSON after running 001 elsewhere.

**Independent Test**: Can be tested by entering Markdown, triggering the
structuring adapter with a mock `ModelProvider`, opening the returned
`StructuringPipelineOutput`, editing one semantic draft field, approving the
required review targets, and exporting an integrated package with immutable base
output, review patches, review decisions, merged reviewed output, source
Markdown, validation report, and `package_status = reviewed_for_structuring`.

**Acceptance Scenarios**:

1. **Given** non-empty Markdown text, **When** the expert chooses the LLM-assisted option and clicks parse, **Then** the workbench sends a `StructuringRunRequest` with `llm_assisted = true` to the Admin structuring adapter and does not call an LLM directly from frontend code.
2. **Given** the adapter returns a valid 001 `StructuringPipelineOutput`, **When** the draft opens, **Then** the workbench displays document metadata, unit tree, semantic draft fields, evidence text, reference candidates, dependency edges, validation findings, and extraction provenance.
3. **Given** the adapter returns validation warnings or LLM schema-validation findings, **When** the draft opens, **Then** the workbench shows those findings as review blockers or warnings instead of hiding them.
4. **Given** the expert edits a draft field, **When** the edit is recorded, **Then** the workbench creates a `StructuringReviewPatch` against the immutable base output and updates only the review session's merged draft state.
5. **Given** the expert reviews document, unit, semantic draft, reference candidate, and dependency edge targets, **When** required targets are approved or explicitly marked as not applicable, **Then** the package may be exported as `reviewed_for_structuring`.
6. **Given** unresolved errors, missing required review decisions, failed adapter execution, or unreviewed required targets, **When** export is requested, **Then** the package remains `needs_review` and explains the remaining review reasons.
7. **Given** export succeeds, **When** the expert inspects the package, **Then** it includes source Markdown, base 001 draft output, review artifacts, merged reviewed output, validation report, extraction provenance, and package status without producing runtime-safe assets or formal rules.

---

### User Story 2 - Import Existing 001 Draft For Review (Priority: P2)

An expert imports an existing 001 `StructuringPipelineOutput` JSON generated by
another tool or previous run, reviews it through the same surfaces, and exports
the same integrated package format.

**Why this priority**: JSON import remains useful for replay, fixture testing,
audit review, and handoff from CLI/batch 001 runs, but it is no longer the main
user path.

**Independent Test**: Can be tested by importing a valid 001 JSON fixture with
LLM-assisted semantic fields, reviewing one target, and exporting the same
integrated package format.

**Acceptance Scenarios**:

1. **Given** a valid `StructuringPipelineOutput` JSON file, **When** the expert imports it, **Then** the workbench validates and opens it as immutable base output.
2. **Given** invalid or schema-incompatible JSON, **When** import is attempted, **Then** no editable review session is created.
3. **Given** the imported draft includes extraction provenance and validation findings, **When** the expert opens it, **Then** those fields are displayed and preserved in export.

---

### User Story 3 - Preserve Future Runtime Preview Boundary (Priority: P3)

Runtime-facing scenario and trace views may remain mocked placeholders, but
they must not consume unreviewed structuring drafts or claim final compliance
results.

**Why this priority**: It preserves the existing 002 boundary while the MVP
focuses on Admin-side structuring review.

**Acceptance Scenarios**:

1. **Given** a runtime placeholder response, **When** it is displayed, **Then** it is labeled as draft/non-final.
2. **Given** a structuring package is `needs_review`, **When** runtime preview attempts to use it, **Then** it is rejected or blocked outside test mode.

## Edge Cases

- Empty Markdown, unsupported files, or adapter failures must show readable
  errors and must not create an editable session.
- Frontend code must not contain API keys or direct model-provider calls.
- LLM schema-validation failures from 001 must be shown as validation findings.
- Missing `parent_unit_id`, `order_index`, `display_label`, hierarchy, source
  location, evidence text, review status, or extraction provenance must be shown
  as incomplete rather than fabricated.
- Reference candidates and dependency edges must be reviewable; hiding them
  would make LLM-assisted output review incomplete.
- Integrated export must preserve immutable `base_output` and apply reviewer
  patches only to `merged_output`.
- A package cannot be marked `reviewed_for_structuring` while required review
  targets remain undecided or validation errors remain unresolved.
- PDF and Word parsing remain out of this feature; Markdown is the direct UI
  input for the MVP.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The workbench MUST accept Markdown regulation/policy text as the primary MVP input.
- **FR-002**: The workbench MUST provide an explicit parse action and an LLM-assisted option that invokes a 001 Admin structuring adapter using `StructuringRunRequest`.
- **FR-003**: The workbench MUST NOT call model providers directly from frontend code; all model calls MUST route through the 001 pipeline and `LLMClient / ModelProvider`.
- **FR-004**: The structuring adapter MUST return either a valid `StructuringPipelineOutput` draft or structured errors; invalid outputs must not create editable review sessions.
- **FR-005**: The workbench MUST display extraction provenance, including `extraction_method`, `prompt_contract_version`, and sanitized `model_trace_id` when provided.
- **FR-006**: The workbench MUST display document classification details, temporal metadata, unit tree, unit text, semantic draft fields, evidence text, reference candidates, dependency edges, validation findings, warnings, and review status.
- **FR-007**: The workbench MUST allow expert edits as `StructuringReviewPatch` records without mutating immutable `base_output`.
- **FR-008**: The workbench MUST allow expert decisions for document, unit, semantic draft, reference candidate, dependency edge, and curation note targets.
- **FR-009**: The workbench MUST maintain a merged draft/reviewed state by applying review patches separately from `base_output`.
- **FR-010**: The workbench MUST export an `IntegratedStructuringReviewPackage` containing source Markdown, source output reference, base output, review patches, review decisions, curation records, merged output, validation report, extraction provenance, package status, and review reasons.
- **FR-011**: `IntegratedStructuringReviewPackage.package_status` MUST be `needs_review` unless all required review gates are satisfied; only then may it be `reviewed_for_structuring`.
- **FR-012**: `reviewed_for_structuring` MUST mean human-reviewed Admin structuring output only; it MUST NOT mean runtime-safe reviewed asset, active `RulePack`, formal `RuleItem`, or final compliance conclusion.
- **FR-013**: Existing 001 JSON import MAY remain as an advanced/replay path, but it MUST NOT be the primary MVP user path.
- **FR-014**: The feature MUST NOT parse PDF or Word files, implement 002 Agent Runtime, retrieval, vector search, rule execution, production asset promotion, final `JudgmentResult`, or compliance conclusion logic.

### Constitution Alignment *(mandatory for Arbiter features)*

- **CA-001**: This is an Admin-side structuring review feature with future Runtime placeholders only; draft and reviewed-for-structuring packages must not enter runtime judgment as formal evidence.
- **CA-002**: Model access is allowed only through the 001 pipeline's `LLMClient / ModelProvider` boundary, with mock-provider test coverage.
- **CA-003**: All handoffs are explicit schema-backed contracts: `StructuringRunRequest`, `StructuringRunResult`, `StructuringPipelineOutput`, review artifacts, and `IntegratedStructuringReviewPackage`.
- **CA-004**: Temporal metadata and extraction provenance must be preserved and shown as incomplete when missing.
- **CA-005**: Evidence text, source locations, reference candidates, dependency edges, and validation findings must remain inspectable.
- **CA-006**: Secrets, raw provider payloads, full prompts, and unnecessary sensitive text must not be displayed or exported by default.

## Key Entities *(include if feature involves data)*

- **MarkdownRegulationInput**: Source Markdown text plus source label and metadata entered in the UI.
- **StructuringRunRequest**: UI-to-Admin-adapter request containing normalized Markdown input, source metadata, run options, and model mode.
- **StructuringRunResult**: Adapter response containing status, `StructuringPipelineOutput` on success, structured errors on failure, sanitized trace summary, and token usage where available.
- **StructuringPipelineOutput**: 001 draft output produced by deterministic and LLM-assisted structuring.
- **ReviewSessionState**: Workbench state containing immutable base output, selected unit, patches, decisions, curation records, review gates, and merged output preview.
- **StructuringReviewPatch**: Field-level expert edit artifact.
- **StructuringReviewDecision**: Target-level expert decision artifact.
- **IntegratedStructuringReviewPackage**: Export package containing source Markdown, immutable base draft, expert review artifacts, merged reviewed output, validation report, provenance, and package status.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An expert can paste Markdown, click LLM-assisted parse, receive a valid 001 `StructuringPipelineOutput`, and open it in the review UI without manually importing JSON.
- **SC-002**: With a mock `ModelProvider`, LLM-assisted classification, semantic draft fields, reference candidates, dependency edges, validation findings, and extraction provenance are visible in the workbench.
- **SC-003**: Expert edits produce review patches and update only merged output, while base output remains unchanged.
- **SC-004**: Exported packages are `needs_review` when required decisions or validation blockers remain.
- **SC-005**: Exported packages may become `reviewed_for_structuring` only after required review gates are satisfied.
- **SC-006**: The feature makes no direct frontend model-provider calls and stores no real secrets in source, fixtures, logs, or exports.
- **SC-007**: The feature produces no active `RulePack`, formal `RuleItem`, runtime-safe reviewed asset, final `JudgmentResult`, or compliance conclusion.

## Assumptions

- 001 remains the canonical owner of structuring schemas and LLM-assisted
  extraction behavior.
- 003 owns the expert workbench UI, review session state, and integrated package
  export.
- A thin Admin structuring adapter is needed because a browser UI cannot invoke
  Python `structure_regulation(...)` directly.
- JSON import remains useful for replay and debugging, but it is not the main
  MVP flow.
